## What we shipped (in plain English)

We built the **foundations of a self-improving engineering system**, packaged as a Claude Code plugin. Think of it as a structured memory and quality gate that lives inside any project you're working on.

Concretely, four things now exist that didn't before:

**1. A typed memory for the project.** Every meaningful piece of engineering work — a goal, a task, a spec, a decision, a bug, an observation, a retrospective — has a place in a small SQLite database (`./.agentic/graph.db`). They're typed, they can link to each other (e.g., "this task implements that goal"), and they survive across Claude Code sessions. The point: future-you, or a future agent, can ask "what do we already know about this area?" instead of starting from scratch.

**2. A specification gate that refuses fuzzy work.** Before any spec can be dispatched, two mechanical checks run:
- Every acceptance criterion must name a real verification — a runnable command like `pytest tests/x.py` or a runtime signal like "logs show zero 5xx errors". Phrases like "works correctly" or "handled appropriately" are rejected outright.
- The spec must include a feedback loop — both an observable signal that something went wrong AND a named fix path. "We'll watch the logs" is rejected because it has no fix path; "if users report bad totals, open a bug and write a retro" passes.

This isn't a style preference. It's an executable check (`validate_spec`) that blocks dispatch.

**3. Two roles with explicit context discipline.**
- The **builder** reads the spec, queries the graph for relevant prior findings in the same scope, implements the work, and records observations back to the graph.
- The **spec-checker** sees only the spec and the artifact — never the builder's commit messages or reasoning. It runs each acceptance criterion's `verify` command exactly as written, marks pass with the captured output as evidence, or logs a Critical finding on fail.

The separation matters because it prevents the all-too-common pattern of "the builder convinces the reviewer with prose instead of evidence."

**4. The connective tissue.** Three slash commands (`/agentic:init` to scaffold a project, `/agentic:detect-conflicts` to surface plugin overlaps without ever modifying another plugin, `/agentic:import-spec` to bridge external plans through the validator), a PowerShell SessionStart hook that walks up from the current directory to find the nearest `.agentic/`, a router skill that orients any subagent, and a spec-writing skill with a 7-question Socratic clarification pass.

## How it works end-to-end, today

Pretend you're starting a new feature in some repo. Here's the actual flow that works right now:

**1. You initialize.** You run `/agentic:init` in the project root. The plugin creates `.agentic/graph.db`, `.agentic/config.json` (with your chosen scope mode), `.agentic/compatibility.json`, and `.agentic/specs/`. Re-running it is safe — nothing gets wiped.

**2. The hook activates the session.** Next time you start Claude Code anywhere inside that project (including subdirectories), the SessionStart hook walks up the directory tree, finds the `.agentic/` you just created, and injects a small context block into Claude's startup: "Agentic Engineering System is active for this project. Project root: ... Open specs: 0. Open critical findings: 0. State lives under .agentic/graph.db..." If there's no `.agentic/` anywhere above your current directory, the hook produces no output at all — it's inert when not in an agentic project.

**3. You write a spec.** You open `templates/spec.md`, copy the structure, fill it in. The `skills/spec-writing/SKILL.md` walks you through the Socratic pass: what changes for the user, what's out of scope, what would falsify "this is done", how would we know if this silently breaks 6 months from now.

**4. The gate runs.** You call `validate_spec` (an MCP tool). If you wrote "verify: tbd" or your feedback loop is "we'll figure it out", the gate returns specific rejection reasons and refuses. You fix and retry. When it passes, you create the Spec node via `create_node(type='Spec', ...)`.

**5. The builder runs.** A builder subagent reads your spec via `get_node`, loads everything the spec lists in `required_reads` via `get_required_reads`, and queries the graph for any open Critical findings in the same scope via `query_graph(type='Finding', scope=..., severity='Critical', status='open')`. It uses that context to plan, then implements test-first.

**6. The spec-checker runs.** A separate spec-checker subagent — which has been given only the spec and the artifact files, NOT the builder's prose — walks each acceptance criterion. For each one: it reads the `verify` field, executes it verbatim, captures stdout+stderr+exit code. On pass, it calls `mark_criterion_satisfied` with the captured output as evidence (the function refuses empty evidence). On fail, it calls `log_finding(severity='Critical', ...)` with the failure detail. It does all criteria — it doesn't stop on the first failure.

**7. The work is recorded.** When everything's green, the Spec node now has all criteria flagged `satisfied: true` with stored evidence, and there's a Strength finding on the graph summarizing the round-trip. When something fails, there's a Critical finding the builder can act on next round.

**8. State survives.** You close the session, come back tomorrow, hit Claude Code in the same directory, and the hook re-injects the project context. The graph remembers everything. The `test_e2e_bootstrap.py` test literally proves this: it runs the full pipeline, closes the SQLite connection, opens a fresh connection, and asserts everything persisted.

## What's NOT in Phase 0 (and why that matters)

The plan was disciplined about scope. Things that look obvious but are deliberately deferred:

- **No code reviewer, no contrarian, no four-tier severity loop.** The spec-checker only logs Critical findings. Style/taste findings (Important, Suggested) are Phase 1.
- **No orchestrator, no parallelism, no git-worktree isolation.** Phase 0 is one builder, one spec-checker, sequential. Phase 2.
- **No pattern-finder, no cross-project pattern memory, no `sqlite-vec` vector index.** The graph schema has slots for Patterns, but there's no agent that produces them yet. Phase 3.
- **No self-improvement loop.** The system doesn't yet update its own validators based on what gets through and later breaks. Phase 4.
- **Windows-only.** The SessionStart hook is PowerShell 5.1. A portable POSIX hook is Phase 1+.

Knowing what's NOT there is as important as knowing what's there — otherwise you'll try to use it as if Phases 1-4 already shipped.

## What surprised me during execution

I logged a few of these in commit messages as inline fixes. None blocked progress, but they're real signals about how to plan Phase 1+.

**The plan's own tests sometimes disagreed with the plan's own validators.** The most obvious case: a test fed the feedback-loop validator the string "If users report incorrect totals, open a bug ticket...", but the validator's signal list contained `"user reports"` and `"user report"` — neither of which is a substring of `"users report"`. So the test failed on first run. Easy fix (added both plural forms to the signal list), but it points at something deeper: when test cases and the code-under-test are written in two separate passes by the same author, they can drift even within a single document. Property-based testing, or just writing tests that exercise the actual phrasing from the worked examples, would catch this.

**A NOT NULL constraint shadowed the test's intent.** One test was supposed to demonstrate that the `CHECK(relation_type IN (...))` constraint rejects garbage relation types. The test's INSERT omitted the `created_at` column, which is `NOT NULL`. So the test was actually proving "NOT NULL works", not "CHECK works". The test still passed (it expected an `IntegrityError`, which both constraints raise), but it wasn't testing the thing it claimed to test. I added the missing column. Lesson: when a constraint test passes, ask "is it passing for the right reason?"

**Second-resolution timestamps + a 10ms sleep = a flaky test.** The graph's `_now()` truncates to seconds via `timespec="seconds"`. The test for "update bumps last_touched" did `time.sleep(0.01)` between create and update, then asserted the new timestamp was strictly greater. On any sub-second clock, that's a coin flip. Bumped to 1.1 seconds. The whole test suite is now ~1 second slower because of one line. Real lesson: timestamp resolution and sleep duration are a pair — they have to be matched.

**PowerShell here-strings ate my commit message.** The plan had me write commit messages as inline `git commit -m "..."` with multi-line bodies. PowerShell 5.1's `@'...'@` heredoc form lost its quoting when piped to a native exe — git received the body as a stream of separate path arguments and bailed with "pathspec '0' did not match any file(s)". I switched to `git commit -F <tempfile>` for every commit after that. The plan's PowerShell snippets that build `python -c @"..."@` strings will hit the exact same problem when anyone runs them verbatim. Worth fixing in the next plan template.

**The plan said "11 tests pass"; actually 12.** Minor planning slip but a real one. A plan that prescribes exact test counts can become a self-imposed bug if it's even slightly wrong, because then "all the tests pass" reads as "wait, I have one extra, what did I do wrong?" Better convention: don't lock in test counts in plans — verify the suite is green, don't audit the cardinality.

**pytest exits 5, not 0, when zero tests are collected.** The Task 0 verify says "exit 0" but the actual pytest behavior with `--collect-only` and zero tests is exit code 5. Easy to misread as a build failure. Plan-side typo.

**The pre-flight gap that the plan missed entirely.** When I sat down to execute, the working directory wasn't a git repo at all. Task 0's Step 9 does `git init && git add . && git commit`. That would have swept up `.engineering-board/last-stop-stdin.json` — a Claude Code session metadata file containing absolute Windows paths and session IDs — straight into a public GitHub repo. I caught it because I checked the dir contents before committing, but the plan itself had no "audit what `git add .` would catch" step. I added `.engineering-board/` to `.gitignore` before the first commit. This is a class of bug that doesn't show up in any test — it shows up six months later when someone greps the public repo for `session_id`. Worth promoting to a standing pre-Task-0 step in Phase 1+ plans.

**Repo creation worked without the scope dance.** Your CLAUDE.md flags that fine-grained PATs need account-level `Administration: Write` for repo creation. Your installed PAT is the classic kind with `repo` scope, and `gh repo create --public --source=.` worked first try. Worth noting in your memory — the warning is real for the fine-grained case but a non-issue here.

**MCP SDK 1.27.1 matched the plan's import shape exactly.** The plan was written against an unspecified SDK version and explicitly flagged that the imports might need to be adjusted. They didn't. Lucky — but a real risk for anyone executing this plan months from now.

## What I learned (lessons for next time)

A few things I'd carry into Phase 1's plan or into my own conventions:

**Plans should ship with the file they describe being in a state where the plan can be replayed.** This plan assumed a git repo existed. It didn't. The very first task had to deal with that fact. Phase 1's plan should explicitly enumerate the precondition state: is the repo initialized, what's on disk that shouldn't be in the commit, are external services authenticated.

**Validators and the tests that exercise them should share fixtures.** The "users report" mismatch was a one-character-class problem that wouldn't have happened if the validator's signal list and the test's input strings had been generated from the same fixtures. Same for the timestamp resolution issue — if `_now()` truncated and the test asserted that update timestamps were `>=` rather than `>` (or used a sleep of N seconds where N matches the resolution), the test wouldn't have been racy.

**"Inline fixes are allowed; note them in commits" was the right contract for this plan.** Every gap I hit had a one-line fix. None justified stopping and re-architecting. The HANDOFF.md explicitly granted that latitude, and that made the execution flow well. Future plans should keep that clause.

**The `.tasks.json` sidecar pulled its weight.** Maintaining a JSON mirror of the native task list felt like overhead the first time I updated it, but it's the only thing that lets the next session pick up where this one stopped without re-running the whole plan. Worth keeping.

**One commit per task was the right granularity.** Each commit reads cleanly in isolation; the diff per task is small; the commit messages tell the story of the build. If I'd batched, the history would be unreadable.

**The exit-gate test paid for itself.** Task 19's e2e test is the single most useful piece of code in the build. It's also the only place where the full pipeline runs as a real user would experience it. Before there's an orchestrator, this test IS the orchestrator. Phase 1's plan should have its own exit-gate test that exercises the new severity loop end-to-end, regardless of how many unit tests pile up around it.

**Phase 1 planning (the actual next chunk of work):**

Phase 1 adds the things deliberately punted from Phase 0. The PRD names them:
- **Code-reviewer and contrarian roles.** New subagents. Spec-checker stays as the falsifiability gate; reviewer adds judgment-based findings (Important, Suggested); contrarian explicitly looks for what the other two might have missed.
- **Four-tier severity loop with critical persistence.** Critical findings need an explicit close-out path; Important/Suggested can stay open as backlog; Strength stays as evidence.
- **Spec-writer subagent.** Right now humans write specs against the template + skill. The skill embeds a Socratic pass; an actual spec-writer agent would run that pass interactively.

**Phase 1's plan-writing itself is the next interesting decision.** Per PRD Gating-4, Phase 0 was built "manually with Claude Code unaided" — the agents we just shipped are deliverables, not used to build themselves. Phase 1 is where you'd start dogfooding: use the Phase 0 spec template + validator + builder + spec-checker to ship Phase 1's pieces. That's the inflection point where the system either starts pulling its weight or you discover the gates are too loose.

**Things worth fixing in the next plan template before Phase 1 starts:**

- Add a pre-Task-0 audit of "what would `git add .` catch that shouldn't ship publicly?"
- Replace inline `git commit -m "<heredoc>"` patterns with `git commit -F <tempfile>` (or `git commit -F -` reading from stdin) for PowerShell safety.
- Don't lock in exact test counts in plan acceptance criteria; just say "suite green".
- Don't specify exit codes for `pytest --collect-only`; just describe the expected stdout.
- When writing a test that exercises a validator's accept-or-reject behavior, draw the input strings from the validator's own fixture set, not from author intuition.

**Long-term (Phases 2-4):** orchestrator + parallelism (Phase 2), pattern-finder + `sqlite-vec` + meta-graph (Phase 3), self-improvement + reviewer calibration (Phase 4). These are months apart; don't pre-plan them yet.
