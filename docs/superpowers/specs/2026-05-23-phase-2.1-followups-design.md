## Item 1 - Retry cap via CriticalLoop reuse

### Problem

Both failure paths reset the task to `pending` and release its claim with no cap, so a
persistently failing task re-dispatches every tick forever:

- NEEDS_FIXING review verdict (`orchestrate.py:220-221`).
- Failed launch / worktree-setup failure (`orchestrate.py:252-257`; setup-failure also
  at `:185-191`).

### Decision

Reuse the Phase-1 CriticalLoop machinery (chosen over a bare counter so the failure
history is durable and queryable in the graph). One shared helper drives BOTH failure
paths. Threshold is the existing `loops.DIAGNOSTIC_THRESHOLD == 3`.

### Task -> loop linkage (survives stateless ticks)

The loop is reached through a dedicated Finding hung off the task:

- Finding: `parent_id == task_id`, `subtype == "dispatch-failure"`, `severity ==
  "Important"`, `status == "open"`. (`log_finding` already takes `parent_id` and copies
  the parent's scope.)
- CriticalLoop: created by `loops.start_critical_loop(conn, finding_id)`; reached later
  via its `finding_id`.

Cross-tick lookup for task `tid`: find the open `dispatch-failure` Finding whose
`parent_id == tid`, then the CriticalLoop whose `finding_id` is that Finding's id. Two
small private query helpers in `orchestrate.py` (direct `SELECT` on the `finding` /
`critical_loop` tables, mirroring the module style: `conn` first, no commit needed for
reads).

### Strike semantics (locked)

`critical_loop.iteration_count` defaults to `1` (schema). `advance_critical_loop` does
`(count or 1) + 1` and stamps `diagnostic_fired_at` when the new count `>= 3`. To make
`iteration_count` equal the cumulative count of FAILED attempts and yield a true
3-strike cap:

- **First failure for a task** (no open dispatch-failure loop exists): create the
  Finding + loop. The loop is born at `iteration_count = 1`, representing this first
  failed attempt. **Do NOT advance.**
- **Each subsequent failure** (open loop exists): `advance_critical_loop` (count
  1->2, then 2->3).
- After create-or-advance, read the loop. If `diagnostic_fired_at` is now set
  (`iteration_count >= 3`, i.e. the 3rd failed attempt): **escalate**. Otherwise:
  **retry** (today's behavior).

So a task is dispatched and fails on attempts 1 and 2 (each resets to `pending`), and
on the 3rd failure it escalates instead of re-dispatching.

### `_handle_failure(conn, tid, claim_id, reason, result)` helper

The caller passes the task's `claim_id` (available at every failure site:
`claim_ids[tid]`, or the local `cid` at the setup-failure site) and the `result` dict.

```
loop = find_open_dispatch_loop(conn, tid)        # None on first failure
if loop is None:
    fid  = findings.log_finding(conn, parent_id=tid, severity="Important",
                                subtype="dispatch-failure", body=reason)
    loop = nodes.get_node(conn, loops.start_critical_loop(conn, fid))
else:
    loop = loops.advance_critical_loop(conn, loop["id"])

if loop["diagnostic_fired_at"]:                   # 3rd strike
    nodes.update_node(conn, tid, status="escalated")
    claims.release_claim(conn, claim_id)          # free the scope; escalated leaves ready set
    result["escalations"].append({"task_id": tid, "reason": reason,
                                   "iterations": loop["iteration_count"]})
else:                                             # retry next tick
    nodes.update_node(conn, tid, status="pending")
    claims.release_claim(conn, claim_id)
```

- NEEDS_FIXING calls it with `reason="NEEDS_FIXING review verdict"`.
- Failed launch calls it with `reason=<the worker error string>`.
- Worktree/setup failure (`:185-191`) calls it too (it currently just appends to
  `failed` and releases the claim - route it through the same helper so it also
  accrues strikes). Note the claim id is available there.

`result["failed"]` keeps listing per-tick failures for visibility (unchanged); the
new `escalations` entries mark the terminal 3rd-strike state.

### Recovery cleanup

On a CLEAN verdict that proceeds to a successful merge, if the task has an open
`dispatch-failure` loop, `loops.resolve_critical_loop(conn, loop_id)` it (and the
Finding can be left open or closed - leave it; the resolved loop is the signal). This
prevents a flaky-then-recovered task from carrying a dangling open loop into future
weeding. Done in the merge step after `status="merged"`.

### Scheduler interaction

No scheduler change needed: `ready_tasks` only surfaces dispatched-spec children whose
deps resolve and that are not already in progress; an `escalated` task is neither
`pending` nor ready, so it drops out of dispatch automatically. (Confirm in a test
that an escalated task is not returned by `ready_tasks`.)

**Acceptance:**
- A task whose launch fails 3 ticks running ends `status="escalated"`, with an open->
  fired CriticalLoop (`iteration_count == 3`, `diagnostic_fired_at` set) and an
  `escalations` entry; it is NOT reset to pending on the 3rd failure and is NOT
  re-dispatched afterward.
- The same holds for 3 NEEDS_FIXING verdicts.
- A task that fails once then succeeds has its dispatch-failure loop resolved and never
  escalates.
- A mix (1 launch-failure + later 2 NEEDS_FIXING, or any interleaving) shares ONE loop
  and escalates on the 3rd cumulative failure.

## Item 4 - Opt-in integration-branch enforcement

`_real_merge` merges each worktree branch into the currently-checked-out branch,
assumed to be the integration branch. Today this is documented, not enforced. Add an
opt-in guard.

### Design

- `tick()` gains `integration_branch: str | None = None`. `None` -> today's behavior
  exactly (assumption stays documented-only; existing callers/tests unaffected).
- New injectable seam `current_branch_fn(repo) -> str`, default `_real_current_branch`
  = `git -C repo rev-parse --abbrev-ref HEAD` (stripped). Injectable so fast tests stub
  it with no real git, consistent with the other seams.
- At the **start of the merge phase** (step 7), if `integration_branch is not None` and
  `current_branch_fn(repo) != integration_branch`:
  - Skip ALL merges.
  - For each CLEAN task, append `{"task_id": tid, "error": "HEAD on <actual>, expected
    integration branch <integration_branch>"}` to `result["escalations"]`, leaving its
    claim held (so a later correctly-branched tick can merge it).
  - Do not mutate task status (they stay whatever the review step left them; CLEAN
    tasks are still `in_progress` with claims held).
- Weeding, ready-set, dispatch, review, and calibration all run normally regardless -
  only the merge phase is guarded. Honors tick()'s never-raise contract.
- CLI: add `--integration-branch NAME` (default unset -> `None`).

**Acceptance:**
- `integration_branch=None`: behavior identical to current tests.
- Matching branch: CLEAN tasks merge normally.
- Mismatched branch: zero merges, every CLEAN task appears in `escalations` with the
  branch-mismatch error, claims remain held, `result["merged"]` is empty.
