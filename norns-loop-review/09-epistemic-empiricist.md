## Highest-leverage asserted-only gaps (load-bearing for project identity)

1. **"The Builder physically cannot read .sealed/"** — the seal is prompt-text + Critic review; `start-loop.ps1:19-25` openly admits the physical seal is "a future hardening target." This is the central anti-reward-hacking claim of the project and it is the weakest in observable enforcement.
2. **"Mutation testing"** — README:26 promises a continuously-running mutation harness; no runner exists. The only references are in `.omc/plans/norns-loop.md:412` (plan-text) and in subagent role descriptions. No `eval_mutation_test.py`, no mutmut config, no CI step.
3. **"PM hard-pivots after 2 failed recoveries"** — no counter, no state file, no enforcement. Charter §6 + `PM_RUNBOOK.md:23,334` are the only references.
4. **"Three Windows Terminal panes side-by-side"** — README:34 directly contradicts `start-loop.ps1:47-63`. Three independent consoles, not wt panes.
5. **"Critic veto routes a task back to pending"** — the routing is described in role prompts (`session-start.ps1:124`) but performed by the Critic session itself. No orchestrator code enforces it; a misbehaving Critic could approve everything and the loop would not detect it (the veto-streak trigger detects too many vetoes, not too few).
6. **Tests append-only via `.sealed/test-hashes.json`** — the hash store is gitignored and absent from this checkout, so the pre-commit append-only gate may be a no-op in the Builder's working tree. No test of the diff-checker.
7. **`.claude/settings.json` deny list is real but unprobed** — none of the deny rules are exercised by a test that runs an actual CC tool call and confirms refusal.

## Solidly grounded claims

- Determinism: `tests/test_cli_snapshot.py`, `tests/test_world_determinism.py`, `tests/test_weight_buffer_determinism.py` all run the deterministic contract and assert byte-equality.
- Network guard: `_netguard.py` is installed and `tests/test_netguard.py` proves an arbitrary host raises `PermissionError`.
- Filesystem-queue cooperation: 45+ envelopes in `governance/queue/done/` plus stale-claim regression test.
- Stuck-state detection: 8+ test cases in `tests/test_orchestrator_stuck_detection.py` exercising all five triggers, including cold-start and window-widening regressions.
- Charter hash check: implemented at both runtime (orchestrator) and commit-time (pre-commit) layers, sharing the same `.charter-lock` artifact.
